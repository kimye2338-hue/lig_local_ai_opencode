# 문서·보고서·슬라이드 디자인 원칙 (생성물 품질 체크리스트)

비서가 보고서/문서/PPT를 만들 때 이 원칙을 적용해 "기본값처럼 밋밋한" 결과가 아니라
읽기 쉽고 전문적인 결과물을 만든다. 회사 문서 톤(간결·근거·한국어)에 맞춘다.

근거: 정립된 디자인/작문 원칙 — Robin Williams의 CRAP(대비·반복·정렬·근접),
Edward Tufte(data-ink, 불필요 장식 제거), 프레젠테이션 통념(슬라이드 1장=1메시지),
효과적인 design doc 구조. (외부 자료 인용은 knowledge/design/_sources.md)

## 공통 원칙 (문서·슬라이드 모두)

- **위계(Hierarchy)**: 제목>소제목>본문의 크기·굵기 차이를 뚜렷이. 한 화면에 강조는 1~2개만.
- **정렬(Alignment)**: 요소를 임의 배치하지 말고 좌측/격자에 정렬. 눈에 보이지 않는 선을 맞춘다.
- **근접(Proximity)**: 관련 항목은 붙이고, 다른 그룹과는 여백으로 분리 — 여백이 곧 구조다.
- **대비(Contrast)**: 강조는 색·굵기·크기 중 하나로 확실히. 애매한 중간 강조는 쓰지 않는다.
- **반복(Repetition)**: 같은 종류엔 같은 서식(불릿 모양, 색, 제목 스타일)을 일관 반복.
- **절제**: 색 2~3개(주색+강조+회색), 폰트 1~2종. 그림자·그라데이션·클립아트 남발 금지(Tufte).
- **한국어 가독성**: 본문 폰트는 맑은 고딕/나눔고딕 계열, 줄간격 넉넉히(1.4~1.6),
  한 줄 너무 길지 않게. 숫자·단위 정렬.

## 보고서/문서 (DOCX/HWP/MD)

- **구조 먼저**: 제목 → 핵심 요약(3줄 이내, 결론 먼저) → 본문(근거/데이터) → 조치/다음 단계.
- **결론 우선(top-down)**: 읽는 사람이 첫 문단에서 "그래서 뭐?"의 답을 얻게.
- **한 문단 한 주제**. 긴 문단은 쪼갠다. 목록으로 낼 수 있으면 목록으로.
- **표**: 헤더 굵게, 숫자 우측정렬, 셀에 설명 문장 넣지 말고 본문에. 격자선 최소.
- **근거 표기**: 수치엔 출처/기준일. 추정은 "추정"이라 명시(우리 상태 어휘와 일관).
- **effective design doc 구조**(기술 문서일 때): 배경/문제 → 목표·비목표 → 제안 → 대안과
  트레이드오프 → 리스크 → 결정. "왜"를 담고 "어떻게"는 필요한 만큼만.

## 슬라이드 (PPTX)

- **1 슬라이드 = 1 메시지**: 제목을 "핵심 문장(주장)"으로 쓴다("매출 현황" X → "3분기 매출 12% 증가" O).
- **글머리 6줄 이내, 한 줄 1행**: 문장 통째로 넣지 말고 키워드. 발표자 노트에 상세.
- **큰 폰트**: 제목 28pt+, 본문 18pt+. 뒤에서도 읽히게.
- **데이터는 차트로**: 표보다 차트, 차트는 축·단위·핵심수치만(불필요 눈금/배경 제거, Tufte).
  강조할 계열만 색, 나머지는 회색.
- **여백 유지**: 슬라이드 가장자리·요소 사이 여백. 꽉 채우지 않는다.
- **일관 템플릿**: 모든 슬라이드 제목 위치·색·폰트 동일. 표지→목차→본문→요약 흐름.

## 구체 수치 규칙 (디자인 시스템 원칙 → 문서/PPT 이식, VoltAgent/awesome-design-md MIT 참고)

- **위계는 4단만**: 제목 → 부제 → 본문 → 캡션. 크기 단계 3~5단 이내(예 PPT: 32~40 / 24 / 18 / 13pt).
  단계를 더 늘리지 않는다.
- **폰트 굵기 최대 2종**: 제목=굵게, 본문=보통. 굵기를 여기저기 남용하면 위계가 무너진다.
- **강조색은 페이지/슬라이드당 1~2곳**만. 본문 글자색은 순검정(#000)보다 **짙은 남색/짙은 회색**
  (예 #0d253d)이 눈이 편하다. 본문/보조/캡션 3단계 색 위계.
- **여백은 한 단위의 배수로**: 기본 단위(예 8pt)를 정하고 그 배수(8/16/24…)로만 여백·문단간격.
  표지·섹션 구분 페이지는 여백 넉넉히, 데이터 밀집 표/그래프 페이지는 여백을 줄인다.
- **숫자 정렬**: 금액/수치 표는 **고정폭 숫자(tabular figures)**로 자릿수를 맞춘다.
- **큰 글자는 자간 살짝 좁게**, 작은 본문은 자간 건드리지 않는다.
- **규칙은 Do/Don't 표**로 간결히 남긴다(스타일 가이드 자체도 이 형식이 읽기 쉽다).

## Office 파일(.xlsx/.docx/.pptx) 작업 운영 원칙

Office 설치 없이 순수 Python(openpyxl/python-docx/python-pptx)으로 만든다
(`agent_ops/office_writer.py`, CLI `report-xlsx`/`office-doc`). 지킬 원칙:

- **기존 템플릿/서식 보존**: 사용자가 준 원본 서식·스타일·수식을 함부로 바꾸지 않는다.
  새로 만들 때도 회사 문서 톤을 유지.
- **수식/데이터 정합**: 엑셀 수식은 오류 0을 목표로. 숫자는 숫자형으로 저장(문자열 금지),
  숫자열은 우측정렬. 합계/평균 등은 실제 값과 일치하는지 확인.
- **출력 검증**: 생성 후 파일이 실제로 열리는지(유효 OOXML) 확인하고, 필요하면
  markitdown 으로 다시 읽어 내용이 맞는지 되짚는다(생성→검증 루프).
- **PPT**: 1 슬라이드 = 1 메시지, 제목은 주장, 글머리 6줄 이내(위 슬라이드 원칙 준수).
- **경로**: 산출물은 `agent_ops/results/reports/` 아래. 원본은 덮어쓰지 말고 새 파일로.

## 생성 시 자기점검 (넘기기 전에)

1. 첫 화면/첫 문단만 봐도 핵심이 전달되는가?
2. 강조가 2개 넘게 흩어져 있지 않은가?
3. 색·폰트·불릿이 일관되는가?
4. 숫자에 출처/기준이 있는가?
5. 슬라이드 제목이 "주제"가 아니라 "메시지"인가?
