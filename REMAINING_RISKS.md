# REMAINING_RISKS

## Security posture

- **AUTO is not "dangerously-skip-permissions".** It only upgrades `ask → allow`
  for bounded, safe, project-local operations (reads, project-local edits, a
  conservative safe-bash allowlist). Explicit `deny` and the command guard
  always win. Network is not auto-opened; external-directory edits are denied;
  credential/OTP/cookie/token extraction is denied; submit/delete/send/upload/
  download style actions remain `ask`.
- **Two independent "auto" concepts exist.** OpenCode already ships a client-side
  `--auto` flag / "Enable auto-approve permissions" command that blanket-approves
  every prompt (in `packages/tui/src/context/permission.tsx` +
  `sync.tsx`). That is a separate power-user escape hatch and is *not* wired to
  this feature. If a user enables `--auto`, it will still blanket-approve
  regardless of the session permission mode. This is pre-existing behavior we did
  not change; document it for users so the two are not confused.

## Implementation risks

- **Mode persistence.** The session mode is in-memory (a `Map` inside the
  `PermissionV2` service), keyed by session. It is **not** persisted across
  server restarts. Persisting it would require a session event + projector
  reducer + DB migration (the session is event-sourced). The TUI re-pushes the
  mode on every prompt (`local.permissionMode.sync`), so after a restart the
  mode is re-established on the next prompt; but tools executed before any prompt
  in a fresh server would default to NORMAL.
- **Control-plane transport.** The TUI sets the mode via a raw `fetch` to
  `PUT /permission/mode/:sessionID` rather than the generated SDK (the SDK is
  code-generated by `@hey-api/openapi-ts`; adding a typed method would require
  regenerating it). The raw call sends the `x-opencode-directory` header +
  `directory` query the same way the SDK does. If a deployment enforces auth
  headers the SDK would otherwise add, the raw call may need those too; for the
  common local `bun run dev` case it works.
- **safe-bash allowlist breadth.** Conservative by design — false negatives only
  cost an extra prompt. Review `SAFE_BASH_LEADERS` / `SAFE_GIT_SUB` /
  `SAFE_PKG_SUB` in `permission/mode.ts` before widening.
- **Command guard heuristics.** The "prose/fake-tool-call mixed into bash"
  heuristics (markdown fences, tool-call markup) are deliberately narrow to avoid
  false positives; they will not catch every adversarial phrasing. The guard is
  a safety net, not a complete sandbox — bash already runs with host authority.
- **Keybind terminals.** Some terminals do not deliver `shift+tab` distinctly;
  `<leader>p` is provided as a reliable fallback, and slash commands always work.

## Build risk

- The full monorepo build/typecheck status is recorded in
