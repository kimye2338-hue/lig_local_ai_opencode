# Current implementation status for Opus review

## Package purpose

This is the current OpenCode AgentOps v3.1 source snapshot plus the latest command-guard/autopilot hardening changes. It excludes heavy one-file installers, embedded-base64 installers, OpenCode binaries, and unrelated large artifacts.

## User environment and hard constraints

- Windows 10 internal-network PC.
- Python 3.11.x is available or expected.
- No external downloads by default.
- BAT files should be ASCII-only and distributed as `.bat.txt`.
- Python files should be UTF-8.
- User prefers single-install workflow when possible, but this review package intentionally excludes heavy installers.
- Company LLM API is OpenAI-compatible but may not support OpenAI native tool_calls perfectly.
- Browser/portal automation must not automate OTP/password/cookie/token extraction.
- Risky portal actions such as approve/submit/delete/send/upload/download must remain blocked unless explicitly approved in the current session.
- User wants robust restart/resume/compaction continuity and not repeated manual prompt fixes.

## Implemented in v3.1 snapshot

- Durable state directory: `agent_ops/state/`
- Task queue: JSONL queue with status, priority, depends_on, touches, retry metadata.
- Checkpoints: `CHECKPOINT.json`, `RESUME_PLAN.md`, `COMPACT_HANDOFF.md`, `LAST_KNOWN_GOOD.json`.
- Interruption detection before heartbeat in status/resume/init path.
- Active zombie task recovery to pending after stale heartbeat detection.
- Stale lock recovery using pid/timestamp in lock files.
- Memory source of truth: `.agent-memory/memory.jsonl`; Markdown files are rendered views only.
- `.agent-memory/*` is not added to opencode.json instructions.
- `memory_manager.recall()` and keyword scoring.
- `llm_plan` injects recalled memory into system prompt.
- Successful tasks are recorded as lessons.
- `/remember` command records user corrections/preferences as high-priority memory.
- External orchestrator loop and parallel loop.
- `touches` field to avoid conflicting parallel tasks.
- `LLM_NOT_CONFIGURED` failure type.
- Experimental `.opencode/plugins/compaction-handoff.ts`.
- Safety policy files are actually read by `safety.py`.

## Latest hardening added after v3.1

- `agentops-autopilot` changed from broad `bash: allow` to guarded bash permissions:
  - safe AgentOps commands allowed;
  - source-file generation through `cat >`, heredoc, long `python -c` denied;
  - ambiguous commands ask.
- `agent_ops/command_guard.py` detects corrupted approval-window commands:
  - reasoning/prose mixed into bash;
  - fake tool-call text;
  - heredoc writing source/config/document files;
  - long echo/printf/python -c;
  - unclosed heredoc;
  - dangerous destructive shell patterns.
- `agent_ops/safe_file_writer.py` provides atomic project-root-only file writing with validation:
  - `.py` py_compile;
  - `.json` parse;
  - `.bat/.cmd` ASCII check.
- `/lintcmd` and `/safecreate` commands added.
- `agentops-command-sentinel` read-only subagent added.

## Known limitations and likely weak points

1. True Claude-Code-like permission-mode toggle was not implemented.
   - Current workaround is separate primary agents: supervisor vs autopilot.
   - Desired: same session/agent with runtime permission mode cycle, ideally Shift+Tab or another keybind.
   - This may require OpenCode source-level patch.

2. Approval modal interception is incomplete.
   - The patch prevents broad autopilot bash auto-execution and denies common bad patterns, but it cannot fully intercept every command generated by OpenCode unless OpenCode core permission resolver or bash tool middleware is patched.
   - Opus should propose a core-level command sanitizer/middleware if possible.

3. Command guard pattern semantics may not match OpenCode exact permission glob/precedence behavior.
   - Verify whether patterns like `"cat > *": deny`, `"* << *": deny`, and `"*<<*EOF*": deny` work in installed OpenCode.
   - If not, propose exact supported syntax.

4. `llm_plan` still produces a text plan/result file; it does not execute the plan in a closed loop.
   - User wants self-maintaining multi-agent improvement, not just plan generation.

5. External parallel orchestrator is basic.
   - Review thread safety, task status races, stale lock recovery, touches conflict logic, and whether writing tasks are truly serialized.

6. Memory recall is keyword-based.
   - Improve relevance without external packages if possible.
   - Consider simple scoring index, recency/priority decay, failure-success pairing, and compaction-safe summaries.

7. Compaction hook support is uncertain.
   - The plugin file exists, but whether the user OpenCode build loads `.opencode/plugins/compaction-handoff.ts` and supports `experimental.session.compacting` is unknown.
   - Need concrete enablement/test instructions or fallback.

8. Portal automation is not complete.
   - Safety gate exists but there is no robust CDP/Selenium portal runner with snapshot schema, selector schema, action quarantine, and strict no-credential/OTP rules.
   - User wants portal research automation eventually, but risky actions must remain blocked.

9. Installer reliability had issues with one-file BAT attempts.
   - Base64 one-file BAT variants hit argument and marker extraction bugs.
   - This package excludes those heavy installers; review lightweight installer only.
   - If Opus can design a safer single-file installer without fragile BAT payload extraction, propose it.

10. There may be source-root assumptions.
   - Current core uses `Path(__file__).resolve().parents[1]` in many files, but review all paths and installer cwd behavior.

## What Opus should especially solve if possible

- Implement or design real permission-mode toggle comparable to Claude Code:
  - Plan / Normal / Autopilot modes;
  - status bar indicator;
  - hotkey cycle such as Shift+Tab if possible without breaking agent cycling;
  - tool permission resolver consults session permission mode before running tools.
- Core-level command sanitizer:
  - before approval modal / before bash execution;
  - block mixed prose+command, heredoc file creation, fake tool calls, long python -c;
  - display actionable fallback.
- A closed self-repair loop:
  - failure -> classify -> create repair task -> repair edits -> verify -> checkpoint -> memory update;
  - without infinite retry or fake success.
- Compaction/restart guarantee:
  - prove hook works or implement fallback.

## Additional review request added

The user also wants Opus to review not only correctness/safety but also high-impact, user-visible feature improvements. The system should feel more like a practical local work assistant, not only a runtime framework.

Opus should identify improvements that make the user experience noticeably better:

- fewer confusing commands/modes;
- simple Korean status and failure messages;
- observable progress;
- automatic repair summaries;
- easier restart/resume flow;
- one-command work/fix/start flows;
- dashboard/timeline/task cards;
- better Windows/internal-network install and diagnostics;
- clear roadmap of improvements ranked by perceived user impact.
